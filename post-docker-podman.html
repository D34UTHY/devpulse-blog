Docker vs Podman em 2025: qual usar no seu workflow?

LC
Lucas Costa
SRE  ·  5 de junho de 2025
👁 3.5k views 💬 24 comentários
🐳

Docker é onipresente. Mas desde 2021, Podman emergiu como alternativa Open Source com arquitetura mais segura — sem daemon único, roda sem root por padrão, compatível com Docker CLI. Em 2025, a questão não é mais "qual é melhor", mas "qual faz sentido pro seu caso".

Resposta curta: Docker pra desenvolvimento, Podman pra sistemas onde segurança importa. Mas vamos aos detalhes.

Arquitetura: o maior diferencial

Docker usa um daemon centralizado (dockerd) rodando como root. Todas as imagens, containers, volumes, e networking passam por esse único processo. É simples, mas é um ponto único de falha.

Podman é daemon-less e rootless-first. Cada usuário pode gerenciar seus próprios containers sem privilégios de root. A arquitetura é distribuída — cada instância do Podman gerencia seus próprios containers.

# Docker: roda como root
sudo dockerd

# Podman: roda como user normal
podman run -d nginx
bash

Implicação de segurança: em Docker, escapar de um container significa root no host. Em Podman, você vira um usuário regular não-privilegiado.

Compatibilidade: Podman copia Docker

Podman foi desenhado como drop-in replacement de Docker. Mesmos comandos, mesma sintaxe:

# Funciona em ambos
podman build -t minha-app .
podman run -p 8080:3000 minha-app
podman push quay.io/seu-usuario/minha-app
bash

A CLI é tão compatível que tem gente que usa alias docker=podman. Mas há diferenças:

  • Docker Compose: Docker tem integrado. Podman precisa de podman-compose (não é tão polido)
  • BuildKit: Docker Buildkit é otimizado. Buildah (alternativa Podman) é bom, mas tem quirks
  • Networking avançado: Docker tem múltiplas redes customizadas de forma elegante. Podman é mais básico

Performance

Em workloads normais, performance é similar. Mas há casos onde divergem:

Docker vence em: networking complexo, multi-container com muitos volumes, build de imagens grandes (BuildKit é mais otimizado).

Podman vence em: containers simples, rootless mode (menos overhead), sistemas embarcados/CI (cgroup v2, melhor isolamento).

Diferença real? 5-15% em casos específicos. Não é decisiva.

DevOps e infraestrutura

Docker domina em sistemas existentes — Kubernetes usa Docker (ou containerd, que é similar). Migrando pra Podman em produção? Requer trabalho:

  • Kubernetes com Podman requer crictl (CLI de baixo nível) — não é automático
  • Monitoramento/observabilidade: ferramentas Docker-nativas podem não funcionar igual
  • Talent pool: mais devs conhecem Docker que Podman

Onde Podman brilha em produção: CI/CD (GitHub Actions, GitLab CI), servidores que não usam Kubernetes, e sistemas embarcados com restrições de privilégio.

Quando usar cada um

Use Docker se:

  • Você usa Kubernetes (Docker é padrão, menos atrito)
  • Trabalha em equipe grande (mais recursos, documentação, tutoriais)
  • Precisa de Docker Compose estável
  • Trabalha com múltiplos registries e networking avançado

Use Podman se:

  • Segurança de containers é crítica (rootless por padrão)
  • Roda em CI/CD (GitHub Actions prefere Podman por segurança)
  • Trabalha em ambiente embarcado / low-resource
  • Quer algo de verdade Open Source (CNCF, não corporate)

Em 2025: convivência saudável

Docker não vai desaparecer. Kubernetes não vai mudar de Docker pra Podman amanhã. Mas Podman cresceu e é uma verdadeira alternativa agora.

A tendência? Containerização sem daemon específico (OCI-compliant). Tanto Docker quanto Podman convergem pra esse padrão. A briga de ferramentas vai importar cada vez menos.

Minha recomendação: desenvolve localmente com o que preferir, mas testa no CI/CD com Podman antes de ir pra produção. Encontra incompatibilidades cedo.
