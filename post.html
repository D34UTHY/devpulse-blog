Rust Back-end Performance · 8 min de leitura

Por que o Rust está conquistando o back-end em 2025

RN
Rafael Nunes
Engenheiro de Software Sênior  ·  12 de junho de 2025
👁 3.2k views 💬 28 comentários 🔖 142 saves
🦀

Durante anos, o back-end foi dominado por Go, Java, Python e Node.js. Cada um com suas vantagens, cada um com seus trade-offs bem conhecidos. Mas desde 2023, algo mudou: Rust está aparecendo cada vez mais em sistemas de produção de empresas sérias — e não só nos "projetos de sistemas" tradicionais.

Discord reescreveu partes críticas do seu serviço de presença em Rust e reduziu uso de memória de 1GB para 64MB. Cloudflare usa Rust no processamento de requests das edges. O Linux Kernel passou a aceitar Rust como segunda linguagem oficial. O Chromium também.

💡

Este artigo assume conhecimento básico de programação back-end. Não é necessário conhecer Rust previamente — vamos construir o entendimento do zero.

O problema que o Rust resolve

Para entender o apelo do Rust, você precisa entender o trade-off clássico que existia antes dele:

  • Linguagens com GC (Java, Go, Python, Node.js): produtividade alta, performance média, segurança de memória garantida pelo Garbage Collector — mas com pausas imprevisíveis e overhead de memória.
  • Linguagens sem GC (C, C++): performance máxima, controle total — mas com uma classe inteira de bugs devastadores: use-after-free, buffer overflow, data races em threads.

Rust propõe uma terceira via: performance de C++ com segurança de memória garantida em tempo de compilação, sem GC e sem runtime overhead.

O sistema de ownership

O coração do Rust é o sistema de ownership. Em vez de um GC em runtime ou gerenciamento manual de memória, o compilador do Rust rastreia, em tempo de compilação, quem "possui" cada dado e quando esse dado deve ser desalocado.

// Em outras linguagens, isso causaria use-after-free
// Em Rust, o compilador rejeita isso em build time:

let s1 = String::from("hello");
let s2 = s1; // ownership transferido para s2

// Erro de compilação: s1 não existe mais
println!("{}", s1); // ← compile error: value borrowed after move
rust

Esse mecanismo — chamado de borrow checker — elimina toda uma categoria de bugs que custam milhões de dólares em patches de segurança por ano em projetos C/C++.

Performance: os números reais

Benchmark apelativo não ajuda ninguém. Vamos olhar casos reais:

"Quando reescrevemos nosso serviço de Read States de Go para Rust, a latência p99 caiu de 5ms para 1ms, e o uso de memória caiu 10x. O resultado foi imediato em produção."

— Discord Engineering Blog, 2020

A explicação técnica é direta: sem GC, não há stop-the-world pauses. Em sistemas de baixíssima latência (jogos, trading, serviços de presença em tempo real), essa previsibilidade é mais valiosa do que throughput médio alto.

O ecossistema em 2025

O maior obstáculo histórico do Rust era o ecossistema imaturo. Isso mudou. Os principais frameworks web estão estáveis e em uso em produção:

# Cargo.toml — as dependências mais comuns em projetos web Rust

[dependencies]
axum       = "0.7"    # framework web (tokio ecosystem)
tokio      = { version = "1", features = ["full"] }
serde      = { version = "1", features = ["derive"] }
sqlx       = { version = "0.7", features = ["postgres"] }
tower      = "0.4"    # middleware e abstrações de serviço
tracing    = "0.1"    # structured logging
toml

Com axum + tokio, escrever um servidor HTTP assíncrono em Rust ficou comparável à experiência de trabalhar com Fastify no Node.js — mas com tipagem mais forte e performance consideravelmente superior.

Quando faz sentido usar?

Rust não é bala de prata. A curva de aprendizado é real — leva semanas até o borrow checker parar de te frustrar. Mas faz muito sentido quando:

  • Você precisa de latência ultra-baixa e previsível (p99 importa mais que média)
  • O serviço processa alto volume de dados e memória é um custo operacional relevante
  • Você está construindo infraestrutura de plataforma (proxies, parsers, serviços de rede)
  • Segurança de memória é um requisito crítico (fintech, sistemas embarcados, segurança)

Para a maioria dos CRUD APIs e aplicações de negócio, Node.js, Go ou Python continuam sendo escolhas pragmaticamente superiores. O custo de onboarding de Rust raramente compensa nesses casos.

Conclusão

Rust não veio para substituir Go ou Python. Veio preencher o espaço que antes era exclusivo de C++ — mas com uma developer experience muito mais civilizada e segurança de memória garantida pelo compilador.

Se você trabalha com sistemas de alta performance, infraestrutura ou serviços de baixa latência em produção, 2025 é um ótimo momento para investir seriamente no aprendizado de Rust. O ecossistema amadureceu, a comunidade cresceu, e as empresas estão contratando.
