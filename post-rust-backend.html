TypeScript 5.5: as features que você precisa conhecer agora

AM
Ana Mello
Engenheira Front-end  ·  8 de junho de 2025
👁 2.8k views 💬 15 comentários
⚡

O TypeScript 5.5 chegou com mudanças que vão além de melhorias incrementais. Desta vez, temos decorators estabilizados, inferência aprimorada de tipos em closures, e otimizações que tornam o compilador até 40% mais rápido em projetos grandes.

Se você ainda está no 5.2 ou 5.3, este upgrade vale a pena — especialmente se trabalha com frameworks que dependem de decorators (NestJS, Angular, TypeORM) ou lida com tipos complexos em bases de código grandes.

Decorators estáveis (finalmente)

Decorators saíram da fase experimental. A sintaxe @ agora funciona sem flags no tsconfig.json, e segue a especificação oficial do TC39 Stage 3.

function log(target: any, key: string) {
  const original = target[key];
  target[key] = function(...args: any[]) {
    console.log(`Chamando ${key} com`, args);
    return original.apply(this, args);
  };
}

class UserService {
  @log
  createUser(name: string) {
    return { id: 1, name };
  }
}
typescript

Antes, era necessário "experimentalDecorators": true. Agora funciona nativamente. Frameworks como NestJS e TypeORM já migraram para a nova sintaxe — se você usa esses projetos, atualizar para 5.5 elimina warnings de deprecação.

Inferência de tipos em closures

Uma das frustrações históricas do TypeScript: closures perdiam contexto de tipo em situações específicas. O 5.5 corrige isso com inferência melhorada de contexto em funções aninhadas:

const users = [
  { id: 1, name: 'Ana' },
  { id: 2, name: 'Bruno' }
];

// Antes: 'item' era inferido como 'any'
// Agora: 'item' é corretamente { id: number; name: string }
const names = users.map(item => item.name.toUpperCase());
typescript

Parece pequeno, mas elimina dezenas de anotações de tipo redundantes em bases de código grandes. O compilador agora é mais esperto em propagar tipos através de chains de .map(), .filter(), e .reduce().

Performance do compilador

O time do TS reescreveu partes do binder e do checker. Resultado: até 40% mais rápido em projetos com +100k linhas. Em monorepos grandes (pnpm workspaces, turborepo), a diferença é visível:

  • Build de um projeto NestJS com 80 módulos: de 12s para 7s
  • Type-checking de um monorepo React com 15 pacotes: de 18s para 11s

A otimização vem principalmente de cache interno melhorado — o compilador reutiliza resultados de type-checking entre builds incrementais com muito mais eficiência.

Breaking changes (poucas, mas importantes)

O 5.5 é conservador em breaking changes, mas tem duas que podem afetar código existente:

  • Strict null checks mais rigorosos em optional chaining — obj?.prop agora diferencia corretamente entre undefined e null
  • Inferência de tipo em const assertions mudou sutilmente — alguns casos que antes inferiam string agora inferem literal types

Na prática, 95% dos projetos atualizam sem modificação. Mas rode tsc --noEmit antes de deployar para produção.

Vale a pena atualizar?

Sim, especialmente se:

  • Você usa decorators (NestJS, Angular, TypeORM, class-validator)
  • Trabalha em projetos grandes onde build time importa
  • Quer eliminar anotações de tipo redundantes em closures

A migração é tranquila. Atualize o pacote, rode os testes, e você já está aproveitando as melhorias. O ganho de performance sozinho já justifica o upgrade.
