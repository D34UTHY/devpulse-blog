JWT não é suficiente: como implementar autenticação de verdade

RN
Rafael Nunes
Engenheiro de Software Sênior  ·  30 de maio de 2025
👁 6.8k views 💬 52 comentários
🔐

JWT é onipresente. Todo tutorial de autenticação começa com JWT. Mas a maioria das implementações que vejo em produção tem falhas críticas: JWTs sem rotação, sem revogação, sem refresh token strategy. Resultado: uma sessão comprometida rouba o acesso indefinidamente.

Este artigo não é "JWT é ruim". JWT é bom como access token de curta vida. O problema é usá-lo como único mecanismo de autenticação.

💡

Vamos cobrir o padrão que empresas sérias usam: JWT de curta duração + refresh tokens + revogação. Exemplos em Node.js/Express.

Por que JWT sozinho não é suficiente

JWT é um token auto-contido. O servidor não precisa consultar banco de dados para validar — basta verificar a assinatura. Isso é ótimo para escalabilidade, mas causa problemas de segurança:

Problema 1: Sem revogação. Uma vez emitido, um JWT é válido até expirar. Se o token for roubado, você não consegue invalidá-lo antes do vencimento. O atacante continua autenticado.

Problema 2: Tempo de expiração vs experiência do usuário. Se você usa JWT de 1 hora, o usuário precisa fazer login de novo a cada hora — horrível. Se usa 30 dias, um token roubado fica válido por 30 dias — catastrófico.

Problema 3: Não há como garantir que o user ainda tem acesso. Se um admin revogar permissões de um usuário, o JWT dele continua válido até expirar. O usuário continua tendo acesso mesmo tendo sido banido.

A solução: JWT + Refresh Token

O padrão seguro é usar dois tipos de token:

  • Access Token (JWT): curta vida (15 minutos), usado em toda requisição. É o JWT que você conhece.
  • Refresh Token: longa vida (7 dias), armazenado seguro no backend. Usado apenas para obter um novo access token.

Fluxo:

// 1. Login — usuário envia credenciais
// POST /auth/login { username, password }

// 2. Server valida, gera dois tokens
const accessToken = jwt.sign(
  { userId: user.id, role: user.role },
  SECRET,
  { expiresIn: '15m' }
);

const refreshToken = jwt.sign(
  { userId: user.id },
  REFRESH_SECRET,
  { expiresIn: '7d' }
);

// 3. Salva refresh token no banco (não no JWT!)
db.saveRefreshToken(user.id, refreshToken);

// 4. Retorna access token no body, refresh token em HTTP-only cookie
res.cookie('refresh_token', refreshToken, {
  httpOnly: true,
  secure: true,
  sameSite: 'strict',
  maxAge: 7 * 24 * 60 * 60 * 1000
});

res.json({ accessToken });

Renovando o access token

Quando o access token expira (15 minutos depois), o frontend chama um endpoint de refresh:

// POST /auth/refresh — headers vazios, browser manda refresh_token automaticamente

const refreshToken = req.cookies.refresh_token;

// 1. Valida assinatura
const decoded = jwt.verify(refreshToken, REFRESH_SECRET);

// 2. Consulta banco — refresh token ainda existe?
const storedToken = db.getRefreshToken(decoded.userId);
if (!storedToken || storedToken !== refreshToken) {
  // Token foi revogado ou não bate
  throw new UnauthorizedError('Refresh token inválido');
}

// 3. Gera novo access token
const user = db.getUser(decoded.userId);
const newAccessToken = jwt.sign(
  { userId: user.id, role: user.role },
  SECRET,
  { expiresIn: '15m' }
);

return res.json({ accessToken: newAccessToken });
javascript

Agora o backend tem controle. Se quiser revogar sessão de um usuário, simplesmente deleta o refresh token do banco.

Revogação: deslogar o usuário

Logout real:

// POST /auth/logout

const refreshToken = req.cookies.refresh_token;
db.deleteRefreshToken(req.user.id, refreshToken);

res.clearCookie('refresh_token');
res.json({ success: true });
javascript

O access token ainda será válido por até 15 minutos — é aceitável, pois é de curta duração. Mas refresh token foi revogado, então não conseguem gerar novo access token.

Melhorias de segurança

Rotation de refresh token: a cada uso de refresh token, gere um novo refresh token:

// Ao renovar access token, também renova refresh token
db.deleteRefreshToken(decoded.userId, refreshToken);
const newRefreshToken = jwt.sign(
  { userId: decoded.userId },
  REFRESH_SECRET,
  { expiresIn: '7d' }
);
db.saveRefreshToken(decoded.userId, newRefreshToken);

// Retorna ambos
res.cookie('refresh_token', newRefreshToken, { ... });
res.json({ accessToken: newAccessToken });
javascript

Se um refresh token vazar, o atacante consegue usá-lo uma única vez — depois disso, seu token fica inválido quando o servidor detecta um novo refresh token sendo usado.

Familia tokens: mantenha histórico de refresh tokens. Se detectar uso de um token "antigo", invalide toda a sessão (token reuse detection):

// Salva histórico
db.saveRefreshToken(userId, {
  token: newRefreshToken,
  family: familyId,  // múltiplos tokens mesma "família"
  issuedAt: new Date()
});

// Ao receber refresh token, valida se é da família correta e está no histórico
const tokenRecord = db.getRefreshToken(userId, refreshToken);
if (!tokenRecord || tokenRecord.family !== expectedFamily) {
  // Possível roubo! Invalida toda sessão
  db.invalidateAllRefreshTokens(userId);
  throw new UnauthorizedError('Possível ataque detectado');
}
javascript

Checklist de autenticação segura

  • ✓ Access token com expiração curta (5-15 minutos)
  • ✓ Refresh token com expiração longa (7-30 dias) armazenado no backend
  • ✓ Refresh token em HTTP-only cookie (não em localStorage)
  • ✓ Logout invalida refresh token no banco
  • ✓ HTTPS obrigatório (cookies only trabalham com HTTPS em produção)
  • ✓ Rotation de refresh token a cada uso
  • ✓ Detecção de token reuse (roubo de tokens)
  • ✓ Rate limiting no endpoint de login (brute force protection)
  • ✓ 2FA para contas críticas (admin, financeiro)

Conclusão

JWT é perfeito como access token. O erro é usá-lo como token permanente. Implemente refresh tokens, guarde-os no backend, e você tem autenticação segura e escalável — o melhor dos dois mundos.
