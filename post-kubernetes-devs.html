Kubernetes para devs: do zero ao primeiro deploy em produção

LC
Lucas Costa
SRE  ·  5 de junho de 2025
👁 4.1k views 💬 32 comentários
☸️

Kubernetes tem fama de complexo. E é — se você tentar aprender tudo de uma vez. Mas para colocar sua aplicação em produção, você precisa de apenas 5 conceitos fundamentais: Pods, Deployments, Services, ConfigMaps e Secrets.

Este artigo não é um curso completo de K8s. É o caminho mais curto entre "tenho uma aplicação containerizada" e "ela está rodando em produção num cluster". Sem operators customizados, sem service mesh, sem nada além do essencial.

💡

Pré-requisito: você já sabe fazer Docker build e rodar containers localmente. Se não sabe, volte um passo — K8s orquestra containers, não os cria do zero.

Os 5 conceitos que importam

Pense no Kubernetes como uma planilha de Excel para infraestrutura. Você declara o que quer (3 réplicas da aplicação, balanceamento de carga, variáveis de ambiente), e o K8s garante que o estado atual corresponda ao estado desejado.

Pod: a menor unidade deployável. Um ou mais containers que rodam juntos e compartilham rede e storage. Na prática, 95% dos Pods têm apenas 1 container.

Deployment: gerencia réplicas de Pods. Você diz "quero 3 instâncias dessa aplicação", e o Deployment garante que sempre haverá 3 rodando, mesmo se uma cair.

Service: expõe Pods para a rede. Como Pods são efêmeros (podem morrer e ser recriados com IPs novos), o Service dá um DNS estável: minha-api.default.svc.cluster.local.

ConfigMap e Secret: guardam configuração. ConfigMap para dados não-sensíveis (URLs de API, feature flags). Secret para credenciais (tokens, senhas).

Primeiro deploy: uma API Node.js

Vamos deployar uma API simples. Você já tem a imagem Docker no registry (Docker Hub, ECR, GCR). Agora precisamos de 3 arquivos YAML:

# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: minha-api
spec:
  replicas: 3
  selector:
    matchLabels:
      app: minha-api
  template:
    metadata:
      labels:
        app: minha-api
    spec:
      containers:
      - name: api
        image: seu-registry/minha-api:v1.0.0
        ports:
        - containerPort: 3000
        env:
        - name: NODE_ENV
          value: "production"
yaml

Deploy com kubectl apply -f deployment.yaml. O K8s cria 3 Pods, cada um rodando sua imagem Docker. Se um Pod crashar, o Deployment automaticamente cria outro.

Expondo com Service

Agora os Pods estão rodando, mas inacessíveis. Precisamos de um Service:

# service.yaml
apiVersion: v1
kind: Service
metadata:
  name: minha-api
spec:
  type: LoadBalancer
  selector:
    app: minha-api
  ports:
  - port: 80
    targetPort: 3000
yaml

kubectl apply -f service.yaml. Em clouds (AWS, GCP, Azure), type: LoadBalancer provisiona automaticamente um load balancer externo. Em clusters locais (minikube, kind), use type: NodePort ou configure Ingress.

ConfigMap e Secrets

Nunca hardcode credenciais no Deployment. Use Secrets:

# secrets.yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque
stringData:
  DATABASE_URL: "postgresql://user:pass@host:5432/db"
yaml

Referencia no Deployment:

env:
- name: DATABASE_URL
  valueFrom:
    secretKeyRef:
      name: db-credentials
      key: DATABASE_URL
yaml

Secrets são encodados em base64 por padrão — não é criptografia. Em produção, use soluções como Sealed Secrets, External Secrets Operator, ou integração com vaults (AWS Secrets Manager, HashiCorp Vault).

Deploy de nova versão

Você fez código novo, buildou minha-api:v1.1.0. Como atualizar sem downtime?

# Edita deployment.yaml, muda a image para v1.1.0
kubectl apply -f deployment.yaml

# Kubernetes faz rolling update automaticamente:
# - Cria 1 Pod com v1.1.0
# - Espera ficar healthy
# - Derruba 1 Pod v1.0.0
# - Repete até todos estarem v1.1.0
bash

Zero downtime, zero config extra. Se a v1.1.0 crashar ao iniciar, o rollout para automaticamente — os Pods antigos continuam rodando.

Logs e debugging

Comandos essenciais para diagnosticar problemas:

# Ver Pods rodando
kubectl get pods

# Logs de um Pod específico
kubectl logs minha-api-7d9f8c-abcde

# Logs em tempo real (tail -f)
kubectl logs -f minha-api-7d9f8c-abcde

# Descrever um Pod (útil quando não sobe)
kubectl describe pod minha-api-7d9f8c-abcde

# Executar comando dentro do Pod
kubectl exec -it minha-api-7d9f8c-abcde -- /bin/sh
bash

Próximos passos

Com esses 5 conceitos, você já consegue deployar aplicações reais em produção. O que fica de fora? Muito:

  • Ingress (roteamento HTTP, múltiplos services sob um domínio)
  • Persistent Volumes (storage que sobrevive ao Pod)
  • Horizontal Pod Autoscaler (escala automática baseada em CPU/memória)
  • Network Policies (firewall entre Pods)
  • Service Mesh (mTLS, observabilidade avançada)

Mas você não precisa disso no dia 1. Comece simples: Deployments + Services + Secrets. Quando bater em limitação real, adicione complexidade. Kubernetes recompensa quem aprende incremental, não quem tenta absorver tudo de uma vez.
