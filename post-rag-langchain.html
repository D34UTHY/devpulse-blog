Construindo um RAG do zero com LangChain e PostgreSQL

PF
Paula Freitas
Engenheira de Software  ·  2 de junho de 2025
👁 5.2k views 💬 41 comentários
🤖

RAG (Retrieval-Augmented Generation) virou o padrão da indústria para dar contexto customizado a LLMs. Ao invés de fazer fine-tuning — que é caro e lento — você injeta documentos relevantes no prompt em runtime.

A arquitetura é direta: (1) embeddings dos seus documentos → (2) armazena num vector database → (3) busca semântica ao receber pergunta → (4) passa os documentos relevantes pro LLM. Vamos implementar isso do zero.

💡

Você vai precisar de: Python 3.10+, PostgreSQL 15+ com extensão pgvector, e uma API key da OpenAI (ou qualquer modelo de embedding compatível).

Setup: PostgreSQL + pgvector

O pgvector adiciona suporte nativo a vetores no Postgres. É mais simples que configurar Pinecone ou Weaviate, e roda onde você já tem Postgres:

-- No PostgreSQL, habilita a extensão
CREATE EXTENSION vector;

-- Tabela pra guardar os chunks de documento
CREATE TABLE documents (
  id SERIAL PRIMARY KEY,
  content TEXT NOT NULL,
  embedding vector(1536),  -- OpenAI ada-002 gera vetores de 1536 dims
  metadata JSONB
);

-- Índice pra busca vetorial eficiente
CREATE INDEX ON documents USING ivfflat (embedding vector_cosine_ops);
sql

O índice ivfflat usa approximate nearest neighbor search — sacrifica precisão perfeita por velocidade. Em bases com milhões de vetores, é essencial.

Gerando embeddings dos documentos

Você tem PDFs, páginas web, docs internos. Primeiro passo: quebrar em chunks pequenos (~500 tokens) e gerar embeddings:

from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.embeddings import OpenAIEmbeddings
import psycopg2

# Conecta no Postgres
conn = psycopg2.connect("postgresql://user:pass@localhost/dbname")
cur = conn.cursor()

# Divide documento em chunks
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50  # overlap pra não perder contexto nas bordas
)
chunks = splitter.split_text(documento_completo)

# Gera embeddings
embeddings_model = OpenAIEmbeddings()
for chunk in chunks:
    embedding = embeddings_model.embed_query(chunk)
    cur.execute(
        "INSERT INTO documents (content, embedding) VALUES (%s, %s)",
        (chunk, embedding)
    )
conn.commit()
python

chunk_overlap=50 é importante — sem isso, informações divididas entre dois chunks ficam inacessíveis. Overlap de 10-15% é padrão.

Busca semântica

Agora vem a mágica: dada uma pergunta, encontrar os chunks mais relevantes:

pergunta = "Como configurar autenticação JWT?"

# Gera embedding da pergunta
query_embedding = embeddings_model.embed_query(pergunta)

# Busca os 3 chunks mais similares
cur.execute("""
    SELECT content
    FROM documents
    ORDER BY embedding <-> %s
    LIMIT 3
""", (query_embedding,))

documentos_relevantes = cur.fetchall()
python

O operador <-> do pgvector calcula distância de cosseno. Quanto menor, mais similar. Esse SELECT retorna os 3 documentos mais próximos da pergunta no espaço vetorial.

Generation: passando pro LLM

Última etapa: monta o prompt com contexto e envia pro GPT:

from langchain.chat_models import ChatOpenAI
from langchain.prompts import ChatPromptTemplate

# Monta contexto
contexto = "\n\n".join([doc[0] for doc in documentos_relevantes])

# Template do prompt
template = ChatPromptTemplate.from_messages([
    ("system", "Você é um assistente técnico. Use o contexto abaixo para responder."),
    ("user", "Contexto:\n{contexto}\n\nPergunta: {pergunta}")
])

# Chama o LLM
llm = ChatOpenAI(model="gpt-4")
mensagens = template.format_messages(
    contexto=contexto,
    pergunta=pergunta
)
resposta = llm(mensagens)

print(resposta.content)
python

O LLM agora tem acesso aos seus documentos internos. Se a resposta estiver nos chunks recuperados, ele vai usar. Senão, vai dizer que não sabe — muito melhor que alucinar.

Otimizações de produção

O código acima funciona, mas em produção você vai querer:

  • Reranking: recupera 20 chunks, usa um modelo de reranking (Cohere, cross-encoder) pra pegar os top 3 realmente relevantes
  • Metadata filtering: adiciona filtros tipo WHERE metadata->>'source' = 'docs-api' antes da busca vetorial
  • Cache de embeddings: gerar embeddings é caro — cacheia perguntas comuns no Redis
  • Chunk smarter, not harder: em vez de tamanho fixo, use splitters conscientes de estrutura (mantém parágrafos inteiros, não quebra no meio de código)

Quando usar RAG vs fine-tuning

Use RAG quando:

  • Seus dados mudam frequentemente (docs internos, base de conhecimento)
  • Você precisa de transparência (dá pra ver exatamente quais documentos foram usados)
  • Orçamento limitado (fine-tuning de GPT-4 custa dezenas de milhares de dólares)

Use fine-tuning quando:

  • Você quer mudar o *comportamento* do modelo (tom, estilo, formato de resposta)
  • Latência importa muito (RAG adiciona overhead da busca vetorial)
  • Seus dados cabem no contexto sem busca (GPT-4 Turbo tem 128k tokens)

Na prática, muitos sistemas usam os dois: RAG pra contexto dinâmico + fine-tuning pra comportamento customizado. Mas comece com RAG — é mais rápido de validar e iterar.
